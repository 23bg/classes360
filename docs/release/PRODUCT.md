# Classes360 — Complete Product Audit

**Stack:** Next.js 15 (App Router full-stack) · Prisma ORM · MongoDB · TypeScript · Tailwind CSS · shadcn/ui · Razorpay · Meta WhatsApp Cloud API

---

## SECTION 1 — PRODUCT OVERVIEW

### What Problem Classes360 Solves

Indian coaching institutes and training centers (JEE/NEET prep, tuition, computer training, skill development) are operationally chaotic. Admissions teams track enquiries in WhatsApp groups or Excel. There is no single system to capture leads from multiple sources, follow them up, admit students, and then collect fees — all in one place.

Classes360 solves this by providing an **institute operations platform** that handles the complete journey: enquiry → follow-up → admission → fee collection, with a student-facing portal for basic communication.

### Target Customer

Coaching institutes and private educational centres in India:
- JEE/NEET coaching
- Tuition centres
- Computer training centres
- Skill development institutes
- Any privately-run educational operation with 1–50 staff

### Primary Workflows

1. A prospective student submits an enquiry via the public institute page → enters the system as a **Lead**
2. Staff follows up, updates lead status, adds notes, and sets follow-up dates
3. Lead is marked **ADMITTED** → automatically creates a **Student** record
4. Student is assigned to a **Course** and **Batch**
5. A **Fee Plan** is created; payments are recorded manually with method and reference
6. Staff monitors outstanding fees and defaulters on the dashboard
7. Institute owner manages team members and subscription billing

### Core Value Proposition

A single-screen operations hub for small-to-medium coaching institutes: lead pipeline + student database + fee tracking + batch management + a public intake page — replacing WhatsApp groups and Excel sheets.

## SECTION 2 — USER ROLES

The system has **four effective roles**. Three are defined in the Prisma `UserRole` enum; one (`MANAGER`) is used in service-layer types.

| Role | Defined In | Write Access | Manage Billing | Add Team Members |
|---|---|---|---|---|
| **OWNER** | DB enum | ✅ | ✅ | ✅ |
| **EDITOR** | DB enum | ✅ | ❌ | ❌ |
| **MANAGER** | Service types | ✅ | ❌ | ❌ |
| **VIEWER** | DB enum | ❌ (read-only) | ❌ | ❌ |

**Permission implementation** (from permissions.ts):

```
WRITE_ROLES = ["OWNER", "EDITOR", "MANAGER"]
canWriteInstituteData → OWNER | EDITOR | MANAGER
canManageBilling      → OWNER only
addTeamMember         → OWNER only
```

**Limitations:**
- No role-based page access control at route level beyond what is checked in API handlers
- `EDITOR` role exists in DB but is not offered as an option when inviting team members (the team service only accepts `MANAGER` or `VIEWER`)
- Dashboard access is blocked entirely when subscription status is `INACTIVE` or `CANCELLED`

**Student Users** are a separate auth context (separate session cookie), not part of the `User` model role structure.

## SECTION 4 — LEAD MANAGEMENT SYSTEM

### Lead Sources

The `source` field is a free-text string (max 80 chars). Sources enter the system via:

1. **Public enquiry form** — form submission at `/i/{slug}` sends `POST /api/v1/public/{slug}/lead`
2. **Dashboard manual entry** — staff creates a lead directly from the dashboard
3. **QR code** — a QR code can encode the public form URL; `source` field captures attribution
4. **Short links** — sharable short links can point to the public enquiry form

### Lead Fields

| Field | Type | Notes |
|---|---|---|
| `name` | String | min 2, max 80 |
| `phone` | String | Indian mobile regex `[6-9]\d{9}`, unique per institute |
| `email` | String? | optional |
| `source` | String? | free text |
| `course` | String? | free text course interest, not a foreign key |
| `message` | String? | notes from lead, max 1024 chars |
| `followUpAt` | DateTime? | next follow-up date |
| `status` | String | see table below |
| `activities` | Embedded array | activity log entries |

### Lead Statuses

From the UI constants in leads/page.tsx/dashboard/leads/page.tsx):

| Status | Meaning |
|---|---|
| `NEW` | Just entered, not yet contacted |
| `CONTACTED` | Staff has reached out |
| `ADMITTED` | Converted — triggers auto student creation |
| `DROPPED` | Lost/not interested |

Statuses are free-form strings in the DB (not an enum), meaning other custom status values can technically be written. The UI enforces the four above.

### Follow-Up System

- `followUpAt` is a date field on the Lead
- Dashboard surfaces **today's follow-ups** and **overdue follow-ups** (leads whose `followUpAt` is past-due and status is not `ADMITTED` or `DROPPED`)
- Activity events `FOLLOWUP_SCHEDULED` and `FOLLOWUP_COMPLETED` are logged in the embedded `activities` array
- **No automated reminders or notifications are sent** — follow-ups are purely dashboard-visible

### Lead Activity Timeline

Stored as an embedded array (`LeadActivity[]`) within the Lead document. Supported activity types:

- `LEAD_CREATED`
- `STATUS_CHANGED`
- `NOTE_ADDED`
- `FOLLOWUP_SCHEDULED`
- `FOLLOWUP_COMPLETED`
- `ASSIGNED_USER_CHANGED`
- `CONVERTED_TO_STUDENT`

API: `GET /api/v1/leads/{id}/timeline`

### Lead → Student Conversion

When a lead's status is set to `ADMITTED`:
1. System checks for duplicate student by phone number in the same institute
2. If none exists, a new `Student` document is created with the lead's name, phone, and email
3. `CONVERTED_TO_STUDENT` activity is logged on the lead
4. The original lead record is **retained** (not deleted)

### Rate Limiting on Public Lead Form

The public enquiry endpoint enforces in-memory rate limiting: `LEAD_RATE_LIMIT_PER_MIN` submissions per IP per minute. Requests over the limit receive a 429 response.

## SECTION 5 — STUDENT MANAGEMENT

### Student Creation

Students can be created two ways:
1. **Automatic** — lead is marked `ADMITTED` (copies name, phone, email from lead)
2. **Manual** — staff creates student directly via `POST /api/v1/students`

Manual creation payload: `{ name, phone, email?, courseId?, batchId?, admissionDate?, fees? }`

### Student Fields (DB model)

| Field | Type | Notes |
|---|---|---|
| `id` | ObjectId | |
| `instituteId` | ObjectId | institute FK |
| `name` | String | |
| `phone` | String | unique per institute |
| `email` | String? | |
| `courseId` | ObjectId? | FK to Course |
| `batchId` | ObjectId? | FK to Batch |
| `admissionDate` | DateTime | defaults to now |
| `portalUsername` | String? | student portal login |
| `portalEmail` | String? | student portal login |
| `portalPasswordHash` | String? | bcrypt hash |
| `portalActive` | Boolean | default true |
| `createdBy` | ObjectId? | staff user ID |
| `deletedAt` | DateTime? | soft delete |

### Student Lifecycle

```
Lead ADMITTED → Student created → Portal credentials set (optional) → Fee plan created → Payments recorded
```

Course and batch can be reassigned. Student portal access can be activated/deactivated. Soft-delete via `deletedAt`.

## SECTION 7 — FEES AND PAYMENTS

### Architecture

The fee system has two layers:

1. **Fee Plan** (`FeePlan`) — one plan per student, stores `totalAmount` and `dueDate`
2. **Fee Installments** (`FeeInstallment`) — individual payment breakdowns with status `PENDING | PAID | OVERDUE`
3. **Payment Record** (`Payment`) — every recorded payment also creates a payment record (for reporting)

### Fee Plan Fields

| Field | Notes |
|---|---|
| `studentId` | FK to Student |
| `instituteId` | |
| `totalAmount` | Float |
| `dueDate` | Optional DateTime |

### Payment Recording

Via `POST /api/v1/fees/{id}/installments` with:
- `amount` (required)
- `date` (optional, defaults to now)
- `note` (optional, max 1024 chars)
- `method` (optional: "CASH", "UPI", "CHEQUE", "BANK_TRANSFER", etc. — free text max 40)
- `reference` (optional, max 120 chars)

Every payment creates both a `FeeInstallment` (status: PAID) and a `Payment` record.

### Fee Defaulters

`GET /api/v1/dashboard/defaulters` — returns a list of students with outstanding `PENDING` or `OVERDUE` installments, showing: student name, phone, course, total fees, total paid, pending amount, due date.

### Payment Listing

`GET /api/v1/payments` supports filters: `from`, `to`, `studentId`, `method`, `limit`.

### Is Razorpay Used for Student Payments?

**No.** Razorpay is used exclusively for **platform subscription billing** (the SaaS fee paid by the institute owner to Classes360). Student fee payments are recorded manually with method + reference — there is no online payment gateway integration for student fees.

## SECTION 8 — PUBLIC INSTITUTE PAGE

### URL Structure

`/i/{slug}` — where `slug` is the institute's unique public identifier.

### API Endpoint

`GET /api/v1/public/{slug}` — returns institute data: name, description, phone, whatsapp, address, timings, logo, banner, social links, Google map link, and courses list.

### Features on Public Page

From the `instituteService.getPublicPage` implementation:
- Institute name, description, contact details, address
- Logo and banner images
- Social links (website, Instagram, Facebook, YouTube, LinkedIn)
- Google Map link
- Timings
- Course listing (courses without `deletedAt`)
- **Enquiry/Lead form** — `POST /api/v1/public/{slug}/lead`

## SECTION 11 — BILLING SYSTEM

### Subscription Plans

Defined in plans.ts:

| Plan | Monthly Price (INR) | User Limit | Tagline |
|---|---|---|---|
| **STARTER** | ₹999 | 1 | For independent owners |
| **GROWTH** | ₹1,999 | 10 | For institutes with admission staff |
| **SCALE** | ₹4,999 | 50 | For larger institutes / organizations |

Both **MONTHLY** and **YEARLY** billing intervals are supported. Each plan × interval maps to a Razorpay Plan ID via environment variables.

### Subscription States

`TRIAL → ACTIVE → INACTIVE → CANCELLED`

- **TRIAL** — 14-day free trial starts at signup
- **ACTIVE** — paid subscription active
- **INACTIVE** — trial expired or subscription lapsed
- **CANCELLED** — cancelled via Razorpay

Dashboard access is allowed for `TRIAL` and `ACTIVE` only.

### Trial System

- 14-day trial created on `POST /api/auth/signup`
- `trialEndsAt` stored on `Subscription`
- `syncTrialStatus` checks on every billing summary fetch and auto-transitions `TRIAL → INACTIVE` if the trial has expired

### Razorpay Subscription Flow

1. Owner calls `POST /api/v1/billing` with `{ action: "create-subscription", planType, interval }`
2. System calls Razorpay API to create a subscription with `notes.instituteId` embedded
3. Razorpay sends webhook events to `POST /api/v1/webhooks/razorpay`
4. Webhook signature is verified with `x-razorpay-signature` header
5. Events handled: subscription activated → status → `ACTIVE`, payment captured → `currentPeriodEnd` updated, subscription cancelled → `CANCELLED`
6. `POST /api/v1/billing/confirm` handles client-side payment confirmation

### User Seat Enforcement

When adding a team member, the system checks current user count against `subscription.userLimit`. If at limit, returns `PLAN_USER_LIMIT_REACHED` (409).

## SECTION 14 — API STRUCTURE

### Request Flow

```
Browser / Public → Next.js Route Handler → Session/Auth check → Service layer → Repository → Prisma → MongoDB
```

All API routes follow the pattern:
1. Read session from `session_token` httpOnly cookie
2. Verify `instituteId` is present on session
3. Optional role-based permission check via `canWriteInstituteData()` or `canManageBilling()`
4. Pass to service layer (validates input with `zod`, enforces business rules)
5. Repository executes Prisma queries
6. Return `{ success: true, data: ... }` or `{ success: false, error: { code, message } }`

Google / GitHub / OAuth login is **not implemented**.
